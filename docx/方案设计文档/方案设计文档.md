## 1. 技术选型对比分析

### 1.1 小程序前端框架选型

#### 三大主流框架对比

| 维度 | **微信小程序原生** | **uni-app (Vue)** | **Taro 4 (React)** |
|------|-------------------|-------------------|---------------------|
| **开发语言** | WXML + WXSS + JS | Vue 2/3 | React 18 + TypeScript |
| **跨端能力** | ❌ 仅微信 | ✅ 14+ 平台 | ✅ 8+ 平台 |
| **性能** | ⭐⭐⭐⭐⭐ 最优 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ (Taro 4原生渲染) |
| **生态/插件** | 微信官方生态 | DCloud插件市场（丰富） | NPM生态 + 社区插件 |
| **TypeScript** | 支持 | 支持 | ✅ 原生深度支持 |
| **状态管理** | 自行选择 | Vuex / Pinia | Redux / Zustand / MobX |
| **UI组件库** | Vant Weapp / WeUI | uView / uni-ui | NutUI / Taro UI |
| **学习曲线** | 中（专属语法） | 低（Vue开发者） | 中（React开发者） |
| **构建工具** | 微信开发者工具 | HBuilderX / CLI | CLI (Webpack / Vite) |
| **社区活跃度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **适用场景** | 纯微信项目 | 多端覆盖、中小型 | 中大型、React技术栈 |

#### 🏆 最终选择：Taro 4 + React + TypeScript

**选择理由：**

1. **技术栈匹配**：你熟悉/倾向 React，Taro 是 React 生态在小程序领域的最佳选择
2. **TypeScript 原生支持**：类型安全对初学者非常友好，IDE 智能提示能大幅减少 bug
3. **Taro 4 性能飞跃**：Taro 4 引入了原生渲染模式，性能接近原生小程序
4. **React 18 支持**：Concurrent Mode、Suspense 等现代特性开箱可用
5. **NPM 生态**：可以直接使用海量 NPM 包，不受限于某个平台的插件市场
6. **未来扩展**：如需扩展到 H5、App（React Native），同一套代码即可

**需要接受的 Trade-off：**
- 跨端平台数量（8个）少于 uni-app（14个），但覆盖了主流平台
- 部分微信原生 API 需要通过 Taro 桥接，偶有兼容性问题

| **LangChain支持** | ✅ 官方首选语言 | ✅ 有JS版本 | ✅ 有JS版本 |
| **自动文档** | ✅ Swagger/OpenAPI 自动生成 | 需手动配置 | ❌ 无 |
| **类型安全** | ✅ Pydantic | ✅ TypeScript | ❌ 原生不支持 |

1. **Python AI 生态无可匹敌**：LangChain、各种 AI SDK 都是 Python 优先，集成最顺畅

4. **Pydantic 数据校验**：请求/响应数据自动校验，减少出错
5. **异步原生支持**：调用 AI API 时天然需要异步，FastAPI 原生支持

### 1.3 AI 模型选型

#### 国内主流大模型 API 价格对比（2025-2026）

| 模型 | 输入价格 | 输出价格 | 特点 |
|------|---------|---------|------|
| **DeepSeek-V3** | ¥1/百万tokens（缓存命中¥0.1） | ¥2/百万tokens | 性价比之王，结构化输出优秀 |
| **DeepSeek-R1** | ¥4/百万tokens（缓存命中¥1） | ¥16/百万tokens | 推理能力强，适合复杂分析 |
| 豆包-pro-32k | ¥0.8/百万tokens | ¥2/百万tokens | 极致低价 |
| 通义千问-Turbo | ¥0.3/千tokens | ¥0.6/千tokens | 稳定可靠 |
| 文心-ERNIE Speed | **免费** | **免费** | 适合开发测试 |
| 智谱 GLM-4 | ¥0.1/千tokens起 | - | 中文能力强 |

#### 🏆 最终选择：DeepSeek

**选择理由：**

1. **价格极具竞争力**：¥1/百万tokens 的输入价格，缓存命中低至 ¥0.1，对独立开发者极其友好
2. **结构化输出能力强**：生成 JSON 格式的题目数据非常稳定，适合刷题场景
3. **OpenAI 兼容 API**：可以直接用 `langchain-openai` 的 `ChatOpenAI` 类对接，零额外学习成本
4. **DeepSeek-R1 推理模型**：生成分析报告时可切换到 R1 模型，推理能力更强
5. **国内访问稳定**：无需翻墙，API 延迟低

**成本估算（MVP阶段）：**
- 每次生成一套题目（约 2000 tokens）：约 ¥0.002
- 每次生成分析报告（约 3000 tokens）：约 ¥0.003
- 日均 100 用户、每人 5 次刷题：约 ¥2.5/天 ≈ ¥75/月
- **MVP阶段几乎可以忽略不计**

### 1.4 LangChain 的角色

LangChain 不是一个"数据库"或"服务器"，而是一个 **AI 编排框架**，它的作用是：

| 功能 | 说明 |
|------|------|
| **Prompt 模板管理** | 把"出题模板""分析报告模板"等结构化管理，方便复用和迭代 |
| **链式调用 (Chain)** | 把"读取用户输入 → 构造 Prompt → 调用 DeepSeek → 结构化输出"串成一条链 |
| **结构化输出** | `with_structured_output()` 直接绑定 Pydantic 模型，大模型 100% 返回合规 JSON，无需手动解析 |
| **流式输出** | 支持 SSE 流式返回，前端可以逐字显示 AI 生成的内容 |
| **后期扩展** | 用户量增长后，可以无缝接入 RAG（检索增强生成）、Agent 等高级功能 |

**简单理解**：LangChain 是 AI 调用的"中间件"，让代码更整洁、更健壮、更好维护。

### 1.5 数据库选型（Phase 2 引入）

| 维度 | **MySQL** | **PostgreSQL** | **MongoDB** |
|------|-----------|---------------|-------------|
| 数据结构 | 关系型，结构化 | 关系型，结构化 | 文档型，灵活 |
| AI数据存储 | 适合用户/题目/记录 | 适合复杂查询 | 适合非结构化数据 |
| 生态 | 最广泛 | 功能最强 | 灵活 |
| 学习资源 | 极其丰富 | 丰富 | 丰富 |
| 云服务 | 各云厂商均支持 | 各云厂商均支持 | Atlas / 各云厂商 |

#### 🏆 最终选择：MySQL

**选择理由：**
1. 用户数据、题目数据、答题记录都是**结构化数据**，关系型数据库最合适
2. 学习资料最丰富，遇到问题最容易找到解决方案
3. 各大云厂商（阿里云、腾讯云）都提供成熟的 MySQL 托管服务
4. 后期可按需引入 Redis 做缓存

### 2.2 MVP 阶段精简架构（无数据库）

**核心理念**：MVP 阶段不引入数据库，所有数据通过 **小程序本地存储** + **AI 实时生成** 来支撑核心业务流程。

```
┌─────────────────────────────────────────┐
│           微信小程序 (Taro 4)             │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │         本地存储 (Storage)         │   │
│  │  - 用户答题历史 (最近 50 条)        │   │
│  │  - 错题缓存 (最近 100 题)          │   │
│  │  - 用户偏好设置                    │   │
│  └──────────────────────────────────┘   │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │         Zustand 状态管理           │   │
│  │  - 当前题目状态                    │   │
│  │  - 答题进度                       │   │
│  │  - 用户交互状态                    │   │
│  └──────────────────────────────────┘   │
└─────────────┬───────────────────────────┘
              │ HTTPS
              ▼
┌─────────────────────────────────────────┐
│       FastAPI 后端 (本地/云服务器)         │
│                                         │
│  POST /api/quiz/generate   → 生成题目    │
│  POST /api/quiz/analyze    → 生成报告    │
│  GET  /api/health          → 健康检查    │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │     LangChain Chain              │   │
│  │  Prompt → DeepSeek → 结构化输出   │   │
│  │  (with_structured_output)        │   │
│  └──────────────────────────────────┘   │
└─────────────┬───────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────┐
│          DeepSeek API                   │
│   V3: 题目生成 / R1: 智能分析            │
└─────────────────────────────────────────┘
```

**MVP 数据存储策略：**

| 数据类型 | 存储位置 | 说明 |
|---------|---------|------|
| 当前答题状态 | Zustand (内存) | 页面刷新即丢失，可接受 |
| 答题历史 | `Taro.setStorageSync` | 本地持久化，最多保留 50 条 |
| 错题记录 | `Taro.setStorageSync` | 本地持久化，最多保留 100 题 |
| 题目内容 | 实时 AI 生成 | 不缓存，每次重新生成（增加题目多样性） |
| 用户设置 | `Taro.setStorageSync` | 学科偏好、难度偏好等 |

### 4.4 成功指标与性能要求

#### API 响应时间指标

| 接口 | 正常耗时 | 超时告警阈值 | 最大容忍 | 说明 |
|------|---------|-------------|---------|------|
| `POST /api/quiz/generate` | 12~20秒 | **60秒** | 90秒 | DeepSeek V3 生成 5 道含选项、长解析的结构化题目，通常需 12~20 秒 |
| `POST /api/quiz/analyze` | 15~30秒 | **60秒** | 90秒 | DeepSeek R1 生成分析报告，推理链更长 |
| `GET /api/health` | <100ms | 1秒 | 5秒 | 健康检查，应极快 |

> **为什么放宽到 60 秒？**
> 在非流式 JSON Mode 下，DeepSeek 需要生成完整的结构化 JSON（包含题目、选项、解析等大量文本），实际测试中 5 道题的生成耗时约 12~20 秒，10 道题可能达到 25~40 秒。将超时告警阈值设为 60 秒可以覆盖大部分正常场景，避免误告警。

### 6.3 Prompt 优化策略

| 策略 | 说明 | 实施阶段 |
|------|------|---------|
| **Few-shot 示例** | 在 Prompt 中加入示例题目，提高输出质量 | Phase 1 |

| **质量校验** | 检查题目数量、选项完整性、答案有效性 | Phase 2 |
| **Prompt 版本管理** | 不同学科使用不同的 Prompt 变体 | Phase 2 |

## 7. 风险评估与应急预案

### 7.1 技术风险

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|---------|
| DeepSeek API 不稳定/超时 | 无法生成题目，核心功能不可用 | 中 | ① 实现自动重试（3次）<br>② 备选 API（豆包/通义千问）<br>③ 本地缓存常用题目 |
| AI 输出格式不规范 | 前端无法渲染题目 | 低 | ① `with_structured_output()` 绑定 Pydantic 模型，100% 返回合规 JSON<br>② 网络异常时自动重试（3次）<br>③ 降级提示 |
| Taro 兼容性问题 | 某些功能在小程序中不生效 | 低 | ① 使用 Taro 官方组件<br>② 及时更新 Taro 版本<br>③ 条件编译处理差异 |
| 小程序审核不通过 | 无法发布 | 低 | ① 遵循微信小程序规范<br>② 不涉及敏感内容<br>③ 提前了解审核要点 |

### 7.2 业务风险

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|---------|
| AI 生成题目质量差 | 用户体验差，留存低 | 中 | ① 持续优化 Prompt<br>② 加入用户反馈机制<br>③ 人工审核样例 |
| AI 生成内容有误 | 知识性错误误导用户 | 中 | ① 加入免责声明<br>② 提供"报告错误"入口<br>③ 关键学科人工校验 |
| API 成本超出预期 | 经济压力 | 低 | ① 控制生成频率<br>② 缓存热门题目<br>③ 设置用户配额 |

### 7.3 MVP 阶段的降级策略

当 AI 服务不可用时，提供基础的降级体验：

1. **显示友好错误提示**："AI 正在休息，请稍后再试"
2. **提供重试按钮**：用户可手动重试
3. **本地错题本仍可用**：即使 AI 不可用，用户仍可复习错题
4. **离线提示文案**：提前准备好静态的提示内容
